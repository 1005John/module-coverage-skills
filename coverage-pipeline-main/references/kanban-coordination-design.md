# Kanban 跨机器协作设计文档

## 1. 问题

覆盖率测试流水线分布在多台机器上，需要双向信息交换：

| 方向 | 信息 | 示例 |
|------|------|------|
| 测试PC → 编译服务器 | 需求 | "DNS 底层需要插桩 cm_async_dns.c" |
| 测试PC → 编译服务器 | 反馈 | "COV_TOTAL_STUBS 太小" |
| 编译服务器 → 测试PC | 交付 | "固件已编译: fw.zip, 76桩" |
| 编译服务器 → 测试PC | 说明 | "cm_plat_dns.c 是死代码" |

目标：完全自动化，两个 Agent 通过共享任务板自主协调，无需人工干预。

## 2. 方案选型

| 方案 | 双向 | 状态管理 | 多机扩展 | 结论 |
|------|------|----------|----------|------|
| 共享文件 + Cron | 手动 | 手动 | 差 | 不够 |
| SSH hermes chat -q | 单向 | 无 | 差 | 无状态 |
| Webhook | 能做 | 无 | 中 | 需 HTTP |
| **Kanban** | **原生** | **自动** | **好** | **✓** |

## 3. 定位：跨技能的协调层

Kanban 不替代现有技能，作为协调层连接它们。执行技能不需要改动。

## 4. Board Schema

### 任务类型

| 任务 | 创建方 | 执行方 | 说明 |
|------|--------|--------|------|
| instrument-{module} | 测试PC | 编译服务器 | 插桩+编译 |
| reinstrument-{module} | 测试PC | 编译服务器 | 根据反馈重插桩 |
| flash-test-{module} | 编译服务器 | 测试PC | 烧录+测试+分析+报告 |

### 生命周期

```
OPEN → CLAIMED → COMPLETED (成功)
                → BLOCKED (失败) → OPEN (解决后重新开放)
```

## 5. 典型交互流程

### 流程 A：新模块测试
1. 测试PC 创建 "插桩DNS" → assignee: build-server
2. 编译服务器 领取 → 插桩+编译 → 完成 → scp 固件
3. 测试PC 检测 → 烧录 → 测试 → 报告

### 流程 B：需要重插桩
1. 测试PC 覆盖率不足 → 创建 "重插桩DNS" → desc: 具体文件和桩ID
2. 编译服务器 领取 → 重新插桩+编译 → 完成
3. 测试PC 再次测试

### 流程 C：编译失败
1. 编译服务器 编译报错 → kanban_block → desc: 错误原因
2. 测试PC 检测阻塞 → 创建修复任务或调整方案

## 6. 实现方案

### 技能结构
放在 coverage-pipeline-main 下作为参考文档，不创建独立技能。

### Cron 配置
每台机器一个 Cron job，1 分钟轮询 board。完整模板见 kanban-cron-templates.md。

### Board 共享
SSH scp 同步 coverage-board.db（与现有固件传输方式一致）。

## 7. 多 PC 扩展
每新增测试电脑：配置 tenant + Cron job，board 自动同步。
