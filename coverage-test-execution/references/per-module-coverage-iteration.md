# 新模块覆盖率迭代测试流程

## 流程

1. **烧录** — 从服务器拉取 release.zip 到测试机，用 adownload.exe 烧录
2. **验证桩识别** — `AT+COVERAGE?` 应显示模块名和总桩数（如 `PING(0%,0%,0/15)`）
3. **执行 AT 命令** — 按功能分组执行，每组后查一次覆盖率
4. **分析增量** — 识别哪些命令贡献了新桩命中
5. **补充边界用例** — 无效参数、边界值、错误路径
6. **迭代直到饱和** — 新增桩为 0 时停止

## 命令分组策略

| 组 | 命令类型 | 覆盖目标 |
|----|----------|----------|
| 1 | Query/Test | GET_CMD / TEST_CMD 分支 |
| 2 | Set (有效参数) | SET_CMD + 参数解析分支 |
| 3 | Error paths | 无效参数 → ERROR 返回 |
| 4 | Boundary | 边界值（0、最大值、-1） |
| 5 | Network (如需) | 涉及网络操作的命令 |

## 每次迭代输出

```
INIT:  PING(0%,0%,0/15)
Group 1 (query/test): PING(33%,0%,2/15)  +2
Group 2 (set):        PING(33%,30%,6/15)  +4
Group 3 (error):      PING(33%,46%,8/15)  +2
FINAL: PING(33%,46%,8/15)
```

## 覆盖率天花板

纯 AT 命令测试通常只能达到 40-50% 分支覆盖率。未覆盖的分支通常是：
- ACTION_CMD 路径（AT 解析器不派发）
- 硬件错误返回路径（ret != 0）
- 条件编译排除的代码（如 ML302A_SUPPORT）
- 需要特定网络状态的分支
