## 关键经验

### 1. 网络依赖是前置条件

DNS/HTTP/MQTT 等网络模块，覆盖率测试前必须检查：
```
AT+CPIN?    → +CPIN: READY
AT+CEREG?   → +CEREG: 0,1 (已注册)
AT+CGACT?   → +CGACT: 1,1 (PDP 激活)
AT+CGPADDR=1 → 有 IP 地址
```
CEREG: 0,0 = 未注册 → 所有 DNS 解析返回 CME ERROR:4 → 成功路径完全不可达。

### 2. 迭代策略顺序

DNS 覆盖率提升的有效顺序：
1. 配置命令全 key set+query（不需要网络）→ 从 27% 到 34%
2. 网络注册后解析成功路径 → 从 34% 到 41%
3. 全 key 的 query 路径（ip/ipv6 查询） → 从 41% 到 49%
4. 主备切换/缓存/超时/重试 → 零新增，确认饱和

### 3. 手册 Note 可能过时

手册说"ML307C 仅支持 priority"，实测固件 4.0.15+ 支持所有 key 的 SET 操作。
**教训**：不要完全依赖手册 Note，实测验证实际支持范围。

### 4. 底层插桩的收益递减

AT 层 49% 饱和后，对底层 cm_async_dns.c 插桩 16 桩（4 stmt + 12 branch）。
结果：branch 从 49%/60桩 降到 39%/76桩 — 分母变大但新桩未被触发。
**教训**：追加底层插桩前，评估新桩是否在 AT 命令可达路径中。

### 5. 死代码发现

cm_plat_dns.c 已插桩 28 桩但不在 .mak 构建系统中，属于死代码。
**教训**：插桩前确认文件在构建系统中。

### 6. AT+COVERAGE=1 在新固件返回 ERROR

新固件可能不支持 AT+COVERAGE=1 清零。脚本应忽略此错误，烧录后覆盖率已归零。
